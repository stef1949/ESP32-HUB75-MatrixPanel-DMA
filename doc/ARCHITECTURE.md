# Architecture Overview

This document describes the internal architecture of the ESP32-HUB75-MatrixPanel-DMA library to help new contributors get oriented quickly.

## High-Level Design

The library drives HUB75 RGB LED matrix panels from an ESP32 using hardware **DMA** (Direct Memory Access), so pixel data is shifted out continuously by the hardware without CPU involvement. Colour depth is achieved through **Binary Code Modulation (BCM)** — each pixel colour is split across multiple bit-planes that are displayed with geometrically weighted durations.

```
Application writes RGB888 pixel
        ↓
updateMatrixDMABuffer()
  • applies CIE 1931 gamma-correction LUT
  • encodes value into BCM bit-planes
        ↓
DMA buffer in SRAM (or OCTAL PSRAM on S3)
        ↓  (hardware, zero CPU)
16-bit parallel GPIO bus → HUB75 connector
```

## Key Data Structures

### `rowBitStruct`
Holds the raw DMA data for **two** physical rows (upper and lower halves are sent simultaneously over R1/G1/B1 and R2/G2/B2). The buffer is `width × colour_depth` `uint16_t` words — one word per clock cycle, one BCM bit-plane stacked after another.

```
data layout:  [plane_0: width pixels] [plane_1: width pixels] ... [plane_N: width pixels]
```

### `frameStruct`
Owns a `std::vector` of `shared_ptr<rowBitStruct>` — one entry per logical row pair. The full frame is therefore a flat list of row buffers, allocated individually to reduce memory fragmentation.

### `HUB75_I2S_CFG`
User-facing configuration structure passed to the constructor. Key fields:
- `mx_width`, `mx_height`, `chain_length` — panel geometry
- `gpio` — 14-pin GPIO mapping (R1 G1 B1 R2 G2 B2 A B C D E LAT OE CLK)
- `driver` / `line_decoder` — shift-register chip variant
- `double_buff` — enable double-buffering
- `i2sspeed` — output clock (8 MHz or 16 MHz)
- `latch_blanking` — OE blanking cycles around LAT pulse (1–4)
- `clkphase` — positive (default) or negative clock edge
- `pixel_color_depth_bits` — BCM depth per channel (2–12, default 8)

## Colour Pipeline

1. **Input**: RGB888 (8-bit per channel) via `drawPixelRGB888()` or 565 via `drawPixel()`.
2. **Gamma correction**: each channel value is looked up in the CIE 1931 LUT (can be disabled with `NO_CIE1931`).
3. **BCM encoding**: for each of the `colour_depth` bit-planes, the corresponding bit of the (gamma-corrected) channel value is ORed into the appropriate `uint16_t` word in `rowBitStruct::data`.
4. **DMA output**: the hardware streams the buffer in a circular loop, spending `2^n` clock cycles per bit-plane `n`, giving binary-weighted exposure times.

## Double Buffering

When `HUB75_I2S_CFG::double_buff = true` (default: false) two `frameStruct` objects are allocated. `flipDMABuffer()` atomically swaps which buffer the DMA hardware reads from and which the CPU writes to. This prevents tearing during animation updates.

## VirtualMatrixPanel

`VirtualMatrixPanel` wraps a `MatrixPanel_I2S_DMA` and remaps logical (x, y) coordinates to physical panel positions. It supports:
- **Horizontal chaining** — panels side by side
- **Grid layouts** — e.g. 2×2 arrangement of 64×32 panels → 128×64 virtual display
- **4-scan panels** — 32×16 panels with four rows updated in parallel

## Build-time Flags

See [`doc/BuildOptions.md`](BuildOptions.md) for the full list. The most commonly used:

| Flag | Effect |
|------|--------|
| `NO_GFX` | Omit Adafruit GFX dependency |
| `USE_GFX_LITE` | Use the lighter GFX_Lite library |
| `NO_CIE1931` | Skip gamma correction |
| `NO_FAST_FUNCTIONS` | Skip optimised line/rect DMA paths |
| `SPIRAM_DMA_BUFFER` | Use OCTAL PSRAM on ESP32-S3 (caps clock to 8 MHz) |
| `PIXEL_COLOR_DEPTH_BITS=N` | Override BCM depth (2–12) |


## Limitations

- **ESP32-C6 / C2 / C3 / H2**: not supported (missing continuous-DMA hardware).
- **Self-PWM panels** (MBI5051, MBI5052, ICN2053, RUL6024, etc.): not supported.
- **PSRAM on non-OCTAL S3**: bandwidth insufficient; use internal SRAM only.
- Memory is the primary constraint; use the [memory calculator](memcalc.md) before targeting large panel chains.
